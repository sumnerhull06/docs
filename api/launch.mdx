The `/launch` endpoint is a browser-facing shortcut. It creates a payment session server-side and immediately renders the Authorize.Net hosted checkout launcher page in the browser â€” no client-side API call or JavaScript fetch is required. This is the endpoint the Chrome extension and the RepairDesk userscript open when a staff member clicks the injected **Authorize.Net Pay** button.

Because the bridge handles session creation itself, this endpoint is safe to open directly from a link, a `window.open` call, or a browser redirect without exposing any API credentials to the client.

## Endpoint

```http
GET /launch
```

## Query parameters

<ParamField query="invoiceId" type="string">
  RepairDesk invoice ID to charge. The bridge tries this value first when resolving which invoice to charge. If neither `invoiceId`, `recordId`, `refId`, nor `ticketId` resolves to a valid invoice, the browser receives an error page.
</ParamField>

<ParamField query="ticketId" type="string">
  RepairDesk ticket ID. The bridge resolves the ticket to its associated invoice server-side. Use this on POS pages where an explicit invoice ID is not available in the URL.
</ParamField>

<ParamField query="recordId" type="string">
  Alternative invoice ID parameter. Treated the same as `invoiceId` â€” the bridge attempts to validate it as a direct invoice ID before falling back to `ticketId`. Useful when the page URL exposes the invoice as `recordId` instead of `invoiceId`.
</ParamField>

<ParamField query="refId" type="string">
  Alternative invoice ID parameter. Treated the same as `invoiceId` and `recordId` â€” the bridge tries `invoiceId`, then `recordId`, then `refId` as direct invoice ID candidates before falling back to `ticketId`.
</ParamField>

<ParamField query="amount" type="number">
  Charge amount override in dollars. Must be greater than zero and must not exceed the current invoice due balance. Defaults to the full due balance if omitted.
</ParamField>

<ParamField query="customerId" type="string">
  RepairDesk customer ID. When provided, the bridge fetches the full customer profile from RepairDesk and uses it to populate all billing fields. Individual `customerEmail`, `customerFirstName`, and other customer parameters take precedence over the fetched profile values.
</ParamField>

<ParamField query="customerEmail" type="string">
  Pre-fills the email field on the Authorize.Net hosted checkout form.
</ParamField>

<ParamField query="customerFirstName" type="string">
  Pre-fills the first name field in the Authorize.Net billing section.
</ParamField>

<ParamField query="customerLastName" type="string">
  Pre-fills the last name field in the Authorize.Net billing section.
</ParamField>

<ParamField query="customerPhone" type="string">
  Pre-fills the phone number field in the Authorize.Net billing section.
</ParamField>

## Invoice ID resolution order

The bridge tries each of the following in order and uses the first value that resolves to a valid RepairDesk invoice:

1. `invoiceId` query parameter
2. `recordId` query parameter
3. `refId` query parameter
4. Invoice derived from `ticketId` (the bridge looks up the ticket and extracts the associated invoice ID)

If none of the supplied parameters resolve to a valid invoice, the browser receives an HTML error page with HTTP status `400`.

## Example URLs

```
GET /launch?invoiceId=12345
GET /launch?invoiceId=12345&amount=49.99
GET /launch?ticketId=67890&amount=52.99
GET /launch?recordId=12345&customerEmail=customer%40example.com
GET /launch?invoiceId=12345&customerEmail=customer%40example.com&customerFirstName=Alex
```

## Response

The bridge returns an **HTML page** containing the Authorize.Net hosted payment form. The page auto-submits the form to Authorize.Net after a 250 ms delay so the customer sees a brief loading state before being taken to the secure card entry form. After the customer completes or cancels payment, Authorize.Net returns them to the bridge's return or cancel URL.

The rendered page also includes:

- The invoice ID and charge amount so the staff member can confirm the correct invoice before payment begins
- A **View sync status** link pointing to the session's `statusUrl` for tracking
- A manual **Continue to secure payment** button in case auto-submit is blocked

## Error behavior

If the bridge cannot resolve an invoice â€” because no valid invoice ID parameter was provided, or because the ticket does not map to a known invoice â€” the browser receives an HTML error page with HTTP status `400`. If RepairDesk or Authorize.Net returns an error during session creation, the browser receives an HTML error page with HTTP status `502`. No JSON is returned from this endpoint under any circumstance.

<Note>
  Use `/launch` for browser-based launch patterns: links, buttons, Chrome extensions, and userscripts. For server-to-server integration â€” for example, generating a checkout link inside your own backend â€” use `POST /api/payments/sessions` instead, which returns JSON including `checkoutUrl` and `statusUrl`.
</Note>
